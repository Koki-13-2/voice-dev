# 提案蓄積機能の根本修正

## 概要

他ディレクトリに対して提案機能を実行した際、提案内容が蓄積されない問題を修正する。

## 問題の根本原因

### 1. `parse_generated`の正規表現が脆弱（最大要因）
- `_GEN_RE`が`^[-*]\s*【`のみ許容
- Claude CLIが番号付きリスト（`1. 【...】`）で出力すると全行パース失敗 → 0件
- 他ディレクトリのCLAUDE.mdが出力形式に影響し、フォーマットが変わりやすい

### 2. ダッシュ文字の不一致
- `—`（em dash）のみ対応
- Claude CLIが`-`（ハイフン）、`–`（en dash）、`ー`（長音）を使うとdescが欠落またはパース失敗

### 3. `result`イベントのテキスト未取得
- `run_oneshot`でsuccessの`result`イベントテキストを無視
- 一部のClaude CLI出力で最終テキストがresultにのみ含まれるケース

### 4. サイレント失敗
- パース0件でも「新しい提案はありませんでした」のみ表示
- デバッグ情報なし

## 変更スコープ

| ファイル | 変更内容 |
|---------|---------|
| `server/proposal_store.py` | `_GEN_RE`正規表現を大幅強化、フォールバックパーサー追加 |
| `server/claude_client.py` | `run_oneshot`で`result`イベントのテキスト取得 |
| `server/main.py` | `_run_propose`にデバッグログ追加、生成テキストをWS通知に含める |
| `client/index.html` | パース0件時にraw出力をトースト表示 |

## 実装方針

### parse_generated の強化
- 番号付きリスト対応: `^\d+[.)]\s*`
- 任意の箇条書き記号: `^[-*•·▸▹►]\s*`
- ダッシュバリエーション: `[—–\-ー]` で統一マッチ
- フォールバック: 正規表現で0件なら`【...】`を含む行を緩くマッチ

### result イベント対応
- `subtype == "success"` かつ `result` フィールドがある場合テキストをyield

### デバッグ可視化
- `_run_propose`でバッファ内容とパース結果件数をログ出力
- WebSocket通知に生成テキスト冒頭を含める
